Architecture Decision Record

ADR-003

Secrets Management Strategy

Status:
Accepted

Context:
This project will handle configuration for AI agent workflows, AWS infrastructure,
and possible third-party integrations. Some values are sensitive and must be managed
as secrets rather than plain configuration.

Decision:
We will manage secrets centrally in AWS Secrets Manager and use IAM roles or other
short-lived identity mechanisms wherever possible instead of long-lived static
credentials.

Non-secret configuration may live in documented environment variables or deployment
configuration, but anything that could expose an account, system, or user data must be
treated as a secret.

Secret names will follow the pattern:

`airbnb-marketing/<environment>/<service>/<secret-name>`

Examples include:

- `airbnb-marketing/dev/bedrock-agent/facebook-api-token`
- `airbnb-marketing/prod/app/database-password`

Rotation will be risk-based. Production secrets should be rotated when supported by the
provider, when access changes, or when compromise is suspected.

Local development will use `.env.example`, ignored local override files, sandbox
credentials, and mocks where possible. Production secrets must never be copied into
local files or committed to the repository.

Reason:
This approach minimizes secret sprawl, keeps credentials out of source control,
supports least-privilege access, and provides a practical path for local development.

Consequences:
- Improved security posture through centralized secret management
- Clear separation between secret and non-secret configuration
- Easier rotation and auditability for sensitive values
- Slightly more setup work for development and deployment workflows
- Need to document how each environment retrieves its secrets

Notes:
- If a future integration supports managed rotation, that should be preferred.
- If a workflow can use IAM or short-lived credentials instead of a stored secret, it
	should do so.
